## **Propuesta: árbol UI escalable para dialogs**

```
app/
  ui/
    app_ui.py                      # entrypoint UI (wiring)
    menu/
      menu_builder.py              # crea barra superior/acciones

    dialogs/
      __init__.py
      base/
        dialog_base.py             # Toplevel + helpers (center, modal, esc)
        form_dialog_base.py        # OK/Apply/Cancel + dirty state
        tabbed_dialog_base.py      # ttk.Notebook + tab helpers
      preferences/
        preferences_dialog.py
        tabs/
          tab_general.py
          tab_llm.py
          tab_ui.py
          tab_advanced.py
      credentials/
        credentials_dialog.py
      registry/
        register_document_dialog.py
        register_collection_dialog.py
      library/
        manage_collections_dialog.py
      chat/
        new_chat_dialog.py
        export_chat_dialog.py
      reports/
        export_report_dialog.py
      common/
        about_dialog.py
        confirm_dialog.py
        error_dialog.py

    widgets/
      __init__.py
      scrollframe.py
      pref_field_factory.py
      labeled_input.py
      masked_entry.py              # para api_keys (mostrar/ocultar)
      status_bar.py
      toast.py                     # opcional: notificaciones

    controllers/
      __init__.py
      preferences_controller.py
      credentials_controller.py
      registry_controller.py
      library_controller.py
      chat_controller.py
      reports_controller.py

    state/
      ui_state.py                  # estado efímero (tab activa, geometry, etc.)
```

### Por qué este árbol funciona

* **`dialogs/base/`**: no repites patrones (modal, centrar, atajos, botones, estado “dirty”).
* **`dialogs/<feature>/`**: cada “dominio UI” tiene su carpeta (preferences, chat, library…).
* **`dialogs/<feature>/tabs/`**: cuando un dialog crece, partes cada tab en su fichero (evitas monstruos).
* **`widgets/`**: componentes reutilizables que no son un dialog completo (entry enmascarado, scrollframe…).
* **`controllers/`**: tu “pegamento” UI→core sin meter lógica de negocio en Tkinter.
* **`menu/`**: la barra superior queda limpia y solo llama controllers.

---

## Convenciones que te ahorran dolores

1. **Un dialog = un fichero**
   Excepto cuando tenga tabs: ahí sí `tabs/`.

2. **El dialog nunca toca JSON directamente**
   Dialog → Controller → Manager (`PreferencesManager`, `CredentialsManager`).

3. **No más de 1 nivel de imports cruzados**

   * `dialogs` puede importar `widgets` y `controllers`
   * `controllers` puede importar `core` (managers)
   * `core` **no importa** nada de UI

4. **Los dialogs devuelven “resultado”**
   Incluso si guardan, que devuelvan un dict tipo:

   ```python
   {"ok": True, "applied": True, "errors": []}
   ```

   Para logs/UI.

## Cómo encaja con tu app completa

* Panel principal: `app_ui.py` con `ttk.Notebook` o layout multi-panel.
* Dialogs: para acciones “puntuales” (config, exportar, crear, importar…).
* Controllers: centralizan decisiones. Por ejemplo:

  * `ChatController.export_chat(...)` abre `ExportChatDialog`
  * `LibraryController.manage_collections()` abre `ManageCollectionsDialog`

## “Minimum viable” para empezar ya

Si quieres ir por ruta mínima, empieza con solo esto:

```
app/ui/dialogs/base/
app/ui/dialogs/preferences/
app/ui/dialogs/credentials/
app/ui/widgets/
app/ui/controllers/
```

Y el resto lo creces cuando haga falta.

## Siguiente paso recomendado (para tu objetivo inmediato)

1. Crear `dialogs/base/tabbed_dialog_base.py`
2. Crear `dialogs/preferences/preferences_dialog.py` con tabs usando ese base
3. Mover/absorber tu harness actual dentro de esa estructura
4. Conectar menú superior “Preferencias…”

Si me dices “vamos”, te genero los **tres ficheros base** (`DialogBase`, `TabbedDialogBase`, `FormDialogBase`) para que todos tus dialogs futuros sigan el mismo patrón.
