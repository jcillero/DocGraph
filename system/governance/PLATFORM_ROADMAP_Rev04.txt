# PLATFORM ROADMAP — DocGraph

**Version:** Rev04
**Nature:** Strategic (Non-Normative)
**Status:** Active
**Aligned with:** Operational Definition Rev09, Invariants Guide Rev05

---

# 1. Purpose

This document defines the **strategic and technical evolution path** of the DocGraph platform.

The roadmap outlines progressive capability development while **preserving the architectural constraints defined in the Operational Definition and the Invariants Guide**.

This document:

* does **not modify architectural invariants**
* does **not alter the frozen MVP baseline**
* does **not introduce undeclared system entities**
* defines **controlled evolutionary phases**

The roadmap exists to ensure that platform evolution remains **architecturally coherent, auditable, and reversible**.

---

# 2. Strategic Evolution Principles

The platform evolves according to the following principles.

### 2.1 Infrastructure Before Intelligence

Core infrastructure must stabilize before introducing advanced reasoning capabilities.

### 2.2 Structure Before Automation

Explicit structures and data models must precede automation or AI integration.

### 2.3 Deterministic Data Before Generative Systems

All generative capabilities must operate on **explicit, traceable, and structured inputs**.

### 2.4 Incremental Capability Without Breaking Invariants

New features must never violate:

* filesystem invariants
* runtime governance rules
* traceability guarantees

### 2.5 Future-Ready Architecture Without Premature Exposure

Capabilities may be **architecturally prepared** but must remain disabled until dependencies are satisfied.

---

# 3. Current Platform Baseline (Consolidated MVP)

The platform currently provides the following capabilities.

### Document Layer

* document registration
* automatic OCR processing
* persistent document identifiers

### Retrieval Layer

* manual collections
* collection-constrained RAG
* deterministic query scope

### Generation Layer

* structured report generation
* reproducible output artifacts

### Governance Layer

* run-linked execution logging
* strict output conventions
* strict system / user / dev separation
* invariant validation

### Interface Layer

* minimal chat-centric user interface
* execution through registered tools

This configuration constitutes the **stable architectural foundation** for future development.

---

# 4. PHASE 1 — Industrial Consolidation

**Objective:** strengthen platform reliability without introducing new functional capabilities.

This phase focuses on **governance, traceability, and operational robustness**.

---

## 4.1 Core Actions

* formalize structured logging
* enforce strict output directory conventions
* integrate minimal smoke tests
* verify absence of hardcoded paths
* validate invariant compliance
* consolidate release-mode packaging

---

## 4.2 Exit Criteria

Phase 1 is completed when the following conditions are satisfied:

* structural audit reports **no critical failures**
* 100% of tools contain valid `SCRIPT_META`
* every execution produces a **traceable run record**
* zero hardcoded paths detected
* release mode fully operational

---

## 4.3 Phase 1 No-Go Rules

During Phase 1 the following are explicitly forbidden:

* introduction of new conceptual entities
* LLM integration
* knowledge graph introduction
* structural UI expansion
* plugin runtime extensions

---

# 5. PHASE 2 — Portable OCR and Local Embeddings

**Objective:** transform the document base into a **reproducible semantic dataset**.

This phase introduces the **semantic processing infrastructure** required for advanced reasoning.

---

## 5.1 Embedded OCR Runtime

A portable OCR runtime will be integrated into the system.

### Dependencies

* stable logging system
* normalized output layout
* persistent document identifiers

### Exit Criteria

* reproducible OCR extraction
* normalized structured text output
* stable document registration workflow

---

## 5.2 Local Embeddings (ONNX or Equivalent)

Vector embeddings will be generated for document chunks.

Embedding models must run **fully offline**.

### Persistence Location

```
user/data/
```

### Dependencies

* stable document identifiers
* stable chunk identifiers
* deterministic chunking strategy

### Exit Criteria

* reproducible vector index
* stable semantic search
* deterministic RAG retrieval

---

## 5.3 Phase 2 No-Go Rules

The following remain excluded:

* local LLM integration
* complex UI extensions
* rule engines
* knowledge graph integration

---

# 6. PHASE 2.5 — Knowledge Graph Layer (RDF / Oxigraph)

**Objective:** represent explicit relationships between system entities.

The knowledge graph introduces **formal semantic structure** to complement vector retrieval.

---

## 6.1 Initial RDF Model

The minimal entity model includes:

* Document
* System
* Requirement
* Modification
* Reference

Relationships must remain **explicit, auditable, and reproducible**.

---

## 6.2 Dependencies

* persistent identifiers
* stable conceptual data model
* strict separation between:

```
vector store
knowledge graph
```

---

## 6.3 Exit Criteria

* reproducible RDF ingestion
* traceable SPARQL queries
* conceptual model integrity

---

## 6.4 Phase 2.5 No-Go Rules

* the knowledge graph must **not replace** the vector store
* rule engines are not introduced
* LLM integration remains postponed

---

# 7. PHASE 3 — Local LLM Integration

**Objective:** enable controlled generation using structured system context.

LLMs will operate **strictly as runtime tools**.

---

## 7.1 Embedded LLM Runtime

The LLM runtime environment will reside in:

```
system/bin/runtimes/llm/
```

Runtime configuration must be declared through the **runtime registry**.

---

## 7.2 Dependencies

* stable vector store
* optional knowledge graph layer
* explicit context construction

---

## 7.3 Exit Criteria

* reproducible LLM responses
* no hardcoded model references
* tool-based execution
* complete run logging linked to `run_id`

---

## 7.4 Phase 3 No-Go Rules

* mandatory cloud dependency
* invariant violations
* hidden prompt injection mechanisms
* bypassing of source attribution

---

# 8. PHASE 3.5 — Causal Analysis Engine (DoWhy)

**Objective:** enable causal reasoning over structured datasets derived from documents.

This capability will be introduced **exclusively through the plugin architecture**.

---

## 8.1 Planned Plugin

Proposed plugin identifier:

```
plugin_id: causal_engine_dowhy
```

Capabilities include:

* causal graph construction
* causal identification
* causal estimation
* causal refutation

---

## 8.2 Dependencies

* reproducible datasets
* stable entity identifiers
* explicit provenance tracking

---

## 8.3 Outputs

Each causal analysis run must generate artifacts including:

* dataset snapshot reference
* causal graph specification
* estimation method
* refutation results
* reproducibility metadata

Artifacts must follow the standard output convention:

```
user/output/<tool_id>/<timestamp>/
```

---

# 9. Plugin Compatibility Requirement

Future system extensions must be implemented through **plugin modules**.

Expected plugin domains include:

* causal inference

**Level 2 — Robust Document Platform**

Industrialized document processing and retrieval.

**Level 3 — Structured Semantic Platform**

Medium-term objective: **Level 3**.

# 15. PHASE 4 — Self-Diagnosis & Structural Intelligence

**Objective:** enable the platform to **analyze its own structure, configuration, and architectural integrity**.

This phase introduces **structural introspection capabilities**, allowing the system to detect inconsistencies, architectural drift, and configuration anomalies automatically.

This phase is particularly important for **portable applications**, where installation environments and runtime conditions may vary.

---

## 15.1 Structural Analysis Tools

The system will provide internal tools capable of analyzing:

* filesystem structure
* configuration registries
* plugin integrity
* invariant compliance
* runtime dependencies

Example analysis domains:

```
registry drift detection
plugin integrity verification
SCRIPT_META completeness
invariant validation
filesystem anomaly detection
```

These diagnostics must generate structured reports stored under:

```
user/runs/system_diagnostics/<timestamp>/
```

---

## 15.2 AI-Assisted System Diagnostics

A specialized tool may use AI to analyze:

* architectural consistency
* dependency anomalies
* configuration conflicts
* evolution risks

The AI diagnostic layer must **never modify the system automatically**.

It may only generate:

* diagnostic reports
* improvement suggestions
* architectural warnings

---

## 15.3 Platform Structural Intelligence

The platform should be capable of building a **structured representation of its own architecture**, including:

* tool registry
* plugin registry
* process catalog
* runtime configuration
* UI architecture
* data model specifications

This structural knowledge base enables:

* automated diagnostics
* architectural documentation generation
* advanced governance tooling

---

## 15.4 Exit Criteria

Phase 4 is considered complete when:

* the platform can generate **self-diagnosis reports**
* registry inconsistencies are automatically detected
* plugin integrity can be validated automatically
* invariant compliance can be evaluated without manual inspection

---

## 15.5 Phase 4 No-Go Rules

The following restrictions apply:

* diagnostic tools must not mutate system state
* diagnostics must not bypass invariant validation
* automated fixes are forbidden unless explicitly approved by the user

---

# 16. PHASE 5 — Autonomous Knowledge Workflows

**Objective:** enable the platform to orchestrate **semi-autonomous knowledge workflows** combining structured data, retrieval, reasoning, and controlled AI generation.

This phase represents the transition from a **document processing platform** to a **knowledge work platform**.

---

## 16.1 Workflow-Driven Knowledge Processing

Workflows will combine multiple capabilities:

```
document ingestion
semantic search
knowledge extraction
context construction
AI-assisted generation
structured output creation
```

Each workflow step must remain **explicit, traceable, and reproducible**.

---

## 16.2 Declarative Knowledge Workflows

Knowledge workflows must be described using **declarative workflow definitions**.

Example conceptual structure:

```
Workflow
 ├ Data Sources
 ├ Retrieval Stage
 ├ Reasoning Stage
 ├ Generation Stage
 └ Output Stage
```

Each stage must correspond to a **registered process type** in the process catalog.

---

## 16.3 AI as Controlled Execution Tool

Artificial intelligence components operate strictly as:

```
registered runtime tools
```

They must:

* produce traceable runs
* log full execution context
* generate reproducible artifacts
* never bypass workflow definitions

---

## 16.4 Human-in-the-Loop Governance

Autonomous workflows must remain **human-supervised**.

Required governance mechanisms include:

* execution approval gates
* traceable workflow execution
* explicit artifact provenance
* reversible workflow steps

This guarantees that AI capabilities remain **controlled and auditable**.

---

## 16.5 Expected Capabilities

When Phase 5 is complete, the platform may support:

* automated report drafting
* structured document analysis pipelines
* causal reasoning workflows
* semantic knowledge extraction
* knowledge graph enrichment
* multi-stage document intelligence workflows

---

## 16.6 Exit Criteria

Phase 5 is considered complete when:

* knowledge workflows can be executed declaratively
* AI tools operate strictly through the runtime registry
* workflow execution produces reproducible artifacts
* full traceability exists between data sources, processes, and outputs
