# OPERATIONAL_DEFINITION

## Portable Workflow Platform

### Revision 09

---

# 1. Purpose

The platform provides a **portable environment for structured knowledge workflows** combining:

* document management
* workflow execution
* notebook interaction
* AI-assisted reasoning
* reproducible project contexts

The system is designed to run **locally and independently of external services**, supporting both human-driven and AI-assisted processes.

The platform is centered on **projects governed by workflows**.

---

# 2. Architectural Principles

The system follows several architectural principles.

### 2.1 Portability

The platform must be deployable as a **portable application**.

Requirements:

* no mandatory external services
* all operational components runnable locally
* projects must remain portable across machines

---

### 2.2 Workspace Separation

The workspace must preserve a strict separation between system layers.

```
system/
dev/
user/
```

Purpose:

* prevent development artifacts from leaking into runtime
* preserve system integrity
* maintain portability

---

### 2.3 Declarative Architecture

Core system components must be defined through **declarative specifications whenever possible**.

Examples:

* workflows
* processes
* UI layouts
* data relationships

Executable modules implement these specifications.

---

### 2.4 Reproducibility

Every significant operation must be traceable.

Workflow executions produce **run records** that allow reconstruction of:

* executed processes
* inputs
* outputs
* generated artifacts

---

# 3. System Overview

The platform architecture can be described as:

```
App Portable
 ├ Process Catalog
 ├ Tool / Action Registry
 ├ Data Layer
 └ Projects
```

Projects contain the operational context for workflows.

---

# 4. Project Model

The **Project** is the primary working unit of the platform.

Each project encapsulates the resources required to perform and reproduce a workflow.

Conceptual structure:

```
Project
 ├ Workflow
 ├ Notebooks
 ├ Sources
 ├ Prompts
 ├ Data
 ├ Artifacts
 ├ Outputs
 └ Runs
```

Projects must remain **self-contained and portable**.

---

# 5. Workflow-Centric Architecture

Each project is governed by a **workflow**.

A workflow defines the operational process applied to project resources.

Structure:

```
Workflow
 ├ Process Instances
 └ Precedence Relations
```

Precedence relations define the execution order between processes.

Constraints:

* precedence relations must form a **Directed Acyclic Graph (DAG)**

---

# 6. Process Model

Processes represent operational steps inside workflows.

A process instance defines:

```
process_id
process_type_ref
inputs
action_ref
outputs
execution_mode
precedence
```

Processes operate on project resources and generate artifacts or outputs.

---

# 7. Process Catalog

Reusable process types are defined in the **system process catalog**.

Location:

```
system/process_catalog/
```

The catalog defines operational building blocks.

Examples:

* ingestion
* analysis
* generation
* validation
* export

Projects instantiate process types rather than redefining them.

---

# 8. Composite Processes

A process type may represent a **composite workflow**.

Composite processes encapsulate internal workflows.

Conceptual model:

```
Composite Process
 └ Subworkflow
     ├ Process
     ├ Process
     └ Process
```

Purpose:

* reuse complex process chains
* simplify project workflows
* maintain modular design

---

# 9. Notebook Interaction Layer

The main interactive interface of the system is the **Notebook Viewer**.

Notebooks allow interaction with:

* documents
* LLM prompts
* workflow results
* analysis tools

Notebooks consist of blocks:

```
Text Block
Prompt Block
Chat Block
Tool Execution Block
Result Block
Visualization Block
```

This structure allows both human and AI-assisted interaction.

---

# 10. Data Layer

The system supports specialized data backends.

### SQLite

Used for structured local persistence.

Typical uses:

* project metadata
* workflow state
* artifact indexing
* run records

---

### Oxigraph

Used as a semantic knowledge graph backend.

Typical uses:

* entity relationships
* semantic document links
* graph queries

---

### Analytical Tools

Analytical tools may be invoked by workflow processes.

Example:

* causal analysis (DoWhy)

These tools operate as **process actions**, not as persistence layers.

---

# 11. Artifacts and Outputs

Artifacts represent intermediate workflow results.

Examples:

* document summaries
* extracted structures
* analysis results

Outputs represent final deliverables.

Examples:

* reports
* generated documents
* exported datasets

Artifacts and outputs must remain structurally separated.

---

# 12. Runs and Traceability

Every workflow execution generates a **Run record**.

Run records capture:

* workflow identifier
* executed processes
* timestamps
* generated artifacts
* outputs

Run records guarantee traceability and reproducibility.

---

# 13. User Interface Model

The interface follows a **project navigation paradigm**.

### Left Panel

```
Project Selector
Project Tree
```

Project tree example:

```
Workflow
Notebooks
Sources
Prompts
Artifacts
Outputs
Runs
```

---

### Main Panel

The main panel hosts the **Notebook Viewer**.

Selected project objects are rendered in this viewer.

---

# 14. Extensibility

The platform supports extensions through:

* process catalog additions
* tool registry
* plugin modules

Extensions may introduce:

* new process types
* new analysis tools
* additional UI components

---

# 15. Governance

The architecture enforces governance through:

* declarative specifications
* invariant validation
* explicit registries
* strict layer separation

This ensures that system behavior remains auditable and maintainable.

---

# 16. Architectural Summary

The platform can be described as:

> A portable system for executing structured workflows over knowledge resources using reusable processes, notebooks and AI-assisted tools.

The architecture combines:

* project encapsulation
* workflow orchestration
* reusable process libraries
* notebook-based interaction
* specialized data layers

This model supports **portable, reproducible and extensible knowledge workflows**.
